1. Introduction

 

The Aristo Technical Support Center (ATSC) has many technicians, which help maintain Aristo College’s computer labs and computer systems. To solve the management issues related to case assignment and lab or equipment reservation handling, ATS Center is implementing a web based tool (ATA) to facilitate problem recording and scheduling the machines and equipment to lab sessions or classes mainly to help staff and faculty members by providing easy access to schedule information and timely problem tracking and assignment. Along with that, the tool can also automatically assign technicians to problems and maintain a record of problem status. Here are some of the features built in the tool:

 

a) Support the faculty/staff to schedule specific machines for class/lab sessions.

b) Enable faculty/staff to report computer-related problems.

c) Record problem reports and schedule technicians to fix the hardware/software problems reported.

d) View the assignment schedules of machines.

 

Following sections outlines the system, technology and functionality of tool.

1.1 System Description

 

            The ATA tool consists of client-side forms, which take information from user and deliver it to the web-server side CGIs. The CGIs parse the information and put the information in the respective files. The username and password is checked for the access control. The department is check to find out whether is request belongs to a supported organization or not. The user sending the request is informed about the status of his problem or given the information about his lab or equipment reservation request online. Tool also assigns a technical assistant to a problem automatically. Following section gives detail about the tools functionality and technologies.

Functionality of the CGI that handles the lab scheduling is as follows:

 

CGI works with following files:

 

a) lab_schedules: which maintains all the schedules; a schedule consists of the group (of machines) id, course allocated, instructor, days, start and end date.

b) machinesDB: which has a list of machines and their groups including a machine id, its operating system and a group it belongs to.

c) userFile: which has the user information such as username, full name, rank or user group i.e. student, staff or faculty, email, office number, phone number and an encrypted password about faculty and staff members.

 

The CGI script checks if the request is valid by verifying that the user name and password are valid as per the user file and also checks whether user belongs to supported department i.e. the user has staff or faculty level permissions in the campus systems. The CGI script also checks for availability of the requested number of machines with the OS requirements. If the machines are not available from a single group, machines are allocated from two different groups provided they are in the same location. After allocating a group of machines script records the information in the “lab_schedules” file. If the request cannot be granted based on the number of students, number of machines available or the OS requirements, user is shown an appropriate message

Functionality of the CGI that handles the lab scheduling is as follows:

 

CGI works with following files:

 

1) problemDB: text file to record the problem reports.

2) techDB: text file that has a list of technicians along with their areas of specialty and number of cases assigned to them.

3) schedulesDB: text file that has a list of machines or equipment reported with problems along with assigned technician’s name.

4) problemLOG: text file to log details about a problem.

 

The CGI checks if the user name and password are valid by comparing these against entries in the user file. It also checks if the user has the right permission i.e. he is a member of the supported staff or faculty organizations. CGI script checks for the departments of the person making the request for validity since ATC only support limited departments (CEN, MEN, SE, EEN, CEN and IS). If the dept is anything other than the ones listed above, script sends a reply back to the problem reporter notifying him that ATSC currently does not support his department. If the credentials are right, script randomly generate a problemId number and associate it with the problem report. It then records the appropriate problem information submitted via the form into the file “problemDB” and dumps the detailed information in file “problemLOG”. The script then assigns a technician to the problem after reading the information from the text files, techDB and problemSchedulesDB and load balancing the assignment among technicians while taking care of category of problem and specialty of the technical assistants. If no category is reported, any technician with any specialty with minimum number of cases assigned is selected. Also if many problems for a specific machine are reported on the same day, the same technician is assigned to all the problems for the same machine. After assigning a technical assistant to a problem, the script sends a message to the client with the problem id number and name of the technician assigned to the problem.

1.4 Limitations

 

1)      Tool does not allow technicians to update the problem case status. A separate tool can be used to do that.

2)      Now session information is taken from first user login to other internal pages. User login in application allows user to read help and view schedule along with launching scheduling lab and reporting problem WebPages. However separate username and password in implemented in the two webpage forms to provide security when user is affecting the server side files.

3)      The username and passwords are sent in clear-text from client to server today. This communication can be encoded by setting form encoding and then decoding the form input in Perl CGI.

4)      No search or filtering options are set in view schedule webpage today. This can be a good feature addition.

 

2. Architecture

 

2.1 Architecture Styles Used:

 

 The ATA tool is built using concepts of both client-server architecture and pipes and filters architecture described as follows.

      Client-Server: ATA web based form a clients which take information from users and transfer that information over web to server side CGI scripts. CGI script then update the data base files after security checks.

      Pipes and Filters: The example of pipes and filter architecture in our tool is “sendmail” where a message is created and after the message content is ready a pipe is opened to sendmail program and message is written on it to be delivered to the user.
